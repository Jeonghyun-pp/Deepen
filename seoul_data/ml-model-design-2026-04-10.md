## 1. 전체 서비스 구조에서 ML이 들어가는 위치

### 1.1 결론부터

이 서비스 전체에서 ML이 필요한 지점은 **딱 두 곳**이다.

- **모델 ①** — 안전 점수 산출 (ML, 학습 필요)
- **모델 ②** — 자연어 리포트 생성 (LLM, 학습 불필요)

나머지는 전부 **공간 쿼리 · 외부 API 호출 · 전통 그래프 알고리즘**으로 해결된다.

### 1.2 플로우 A: 주소 → 안전 점수 리포트

```
사용자: "봉천동 OO길 12"
      │
      ▼
[Step 1] 주소 → 좌표 변환          ❌ ML 아님 (카카오맵 API)
      │
      ▼
[Step 2] 반경 500m 공공데이터 수집  ❌ ML 아님 (PostGIS)
      │
      ▼
[Step 3] 원시 데이터 → feature 벡터  ❌ ML 아님 (집계·계산)
      │
      ▼
[Step 4] feature → 안전 점수         ✅ 모델 ① (LightGBM)
      │
      ▼
[Step 5] 점수 + 수치 → 자연어 리포트 ✅ 모델 ② (Claude API)
      │
      ▼
[Step 6] 화면 렌더링                 ❌ ML 아님 (React)
```

6단계 중 4·5단계만 AI.

### 1.3 플로우 B: 출발지·도착지 → 안전 경로

```
사용자: "강남역 → 봉천동 OO길 12"
      │
      ▼
[Step 1] 주소 → 좌표                 ❌ ML 아님
      │
      ▼
[Step 2] 보행자 도로 그래프 로드      ❌ ML 아님 (OSMnx)
      │
      ▼
[Step 3] 각 도로 엣지에 안전 점수    ✅ 모델 ① 재사용
      │
      ▼
[Step 4] 안전 가중 최단 경로 탐색    ❌ ML 아님 (Dijkstra)
      │
      ▼
[Step 5] 상위 3개 경로 추출          ❌ ML 아님 (Yen's K-shortest)
      │
      ▼
[Step 6] 지도 위 경로 렌더링          ❌ ML 아님
```

6단계 중 Step 3 하나만 ML, 그것도 **모델 ①을 재사용**.

> ★ **Baseline 보강**: 서울시 안심귀갓길 경로 데이터(OA-21695, 362개)가 존재하므로, Step 4에서 Dijkstra cost 함수에 "안심귀갓길 구간이면 비용 할인" 가산점을 부여해 **서울시 공식 지정 경로를 우선 활용**. "서울시가 이미 지정한 경로를 기반으로 사각지대는 모델이 보완"이라는 발표 스토리 확보.

### 1.4 통합 아키텍처

```
          [공공데이터 소스]
                 │
                 ▼
        [데이터 수집·전처리 레이어]   ❌ ML 아님
                 │
       ┌─────────┴─────────┐
       │                   │
       ▼                   ▼
  [주소 기반 feature]  [도로 엣지 기반 feature]
       │                   │
       ▼                   ▼
  ╔══════════════════════════════════╗
  ║  모델 ①: 안전 점수 모델            ║
  ║  (LightGBM, 플로우 A·B가 공유)    ║
  ╚══════════════════════════════════╝
       │                   │
       ▼                   ▼
  [점수 + 수치]       [점수가 붙은 도로 그래프]
       │                   │
       ▼                   ▼
  ╔═══════════════╗   [Dijkstra / A*]  ❌ ML 아님

       ▼
  안전 리포트 화면
```

## 2. 왜 경로 탐색은 전통 알고리즘인가

"AI 대회인데 경로도 ML로 해야 하지 않나?"에 대한 답.

| 기준 | Dijkstra / A* | ML 기반 라우팅 (GNN/RL) |
|---|---|---|
| 최적성 | ✅ 수학적으로 최적 보장 | ❌ 근사해 |
| 학습 데이터 | ✅ 필요 없음 | ❌ 대량 필요 (존재 안 함) |
| 속도 | ✅ 밀리초 | ❌ 수백 ms~수 초 |
| 설명 가능성 | ✅ 완벽 | ❌ 블랙박스 |
| 5주 내 구현 | ✅ 1일이면 동작 | ❌ 불가능 |
| 심사 설득력 | ✅ "검증된 알고리즘 활용" | ⚠️ 왜 필요한지 설명 어려움 |

**핵심 통찰**: 이 문제의 진짜 어려움은 "각 도로 구간에 얼마나 위험도를 줄 것인가"이지, "점수가 매겨진 그래프에서 최단 경로를 찾는 것"이 아니다. 앞쪽만 ML(모델 ①)로 풀면, 뒤쪽은 이미 풀려 있는 문제다.

**발표용 한 문장**:

> "AI는 '판단'을 담당하고, 고전 알고리즘은 '탐색'을 담당하는 역할 분담 구조."

## 3. 모델 ② — 생성형 AI 리포트 (쉬운 쪽)

### 3.1 요구사항

| 항목 | 내용 |
|---|---|
| 학습 데이터 | ❌ 불필요 |
| Fine-tuning | ❌ 안 함 |
| Few-shot 예시 | 3~5개 (팀이 직접 작성) |
| 프롬프트 템플릿 | 1개 |
| 모델 | Claude Sonnet 4.5 (품질) 또는 Haiku 4.5 (비용) |
| 추가 기법 | Structured Output (JSON) 필수, RAG 선택 |

### 3.2 Inference 입력 구조 (JSON 예시)

```json
{
  "address": "관악구 봉천동 OO길 12",
  "total_score": 72,
  "category_scores": {
    "cctv": 85, "lighting": 62, "crime": 68,
    "shelter": 71, "nightlife": 78, "road": 70
  },
  "raw_features": {
    "cctv_count_500m": 12,
    "nearest_convenience_min": 8,
    "nearest_police_min": 5,
    "crime_5_trend_yoy": -0.12,
    "streetlight_gaps": 3
  },
  "time_context": {
    "current_hour": 22,
    "sunset_time": "19:12",
    "weather": "clear"
  },
  "alternatives": [
    {"name": "신림동 OO로", "score": 78, "rent_range": "45~55"}
  ]
}
```

### 3.3 주의점

- **Hallucination 방지**: LLM이 숫자를 꾸며내지 않도록 반드시 "제공된 JSON의 값만 참조, 계산 금지" 프롬프트 삽입
- **출력 검증 로직**: LLM 출력에서 숫자를 추출해 입력 JSON과 일치 여부 검사 → 불일치 시 재생성

### 3.4 작업량

**1명 × 2일**. 프롬프트 설계 + few-shot 예시 3~5개 작성 + 검증 로직.

### 4.3 라벨 설계 (정답 y) — 가장 어려운 부분

**문제**: "이 지점의 진짜 안전 점수"를 알려주는 정답이 세상에 없다.

#### 옵션 비교

| 옵션 | 방법 | 장점 | 단점 |
|---|---|---|---|
| A. Proxy | 범죄 발생 건수 (log) | 대량, 객관적 | 신고된 범죄만, 역인과 |
| B. Expert | 사람이 1~5점 라벨링 | 직관과 일치 | 주관적, 샘플 작음 |
| C. **Hybrid** | A로 학습, B로 검증 | 양쪽 장점 | 라벨링 작업 필요 |

**결정**: **옵션 C (Hybrid)**.

- **Training set**: 범죄 proxy label (대량, ~240,000 rows)
- **Validation set**: Expert label 500개 (팀원 5명 × 100개)
- **평가 metric**: 모델 출력 vs expert score의 **Spearman correlation**

**발표 포인트**: "학습은 proxy, 검증은 전문가 판단" — 프로페셔널한 설계로 어필.

### 4.4 샘플링 단위 — "한 행(row)"이 뭔가

| 옵션 | 샘플링 단위 | 샘플 수 | 결정 |
|---|---|---|---|
| 격자 | 100m × 100m | ~60,000 | ✅ **학습용** |
| 행정동 | 동 단위 | ~425 | ❌ 너무 적음 |
| 주소 | 건물 | 수백만 | ❌ 중복 과다 |
| 도로 엣지 | 도로 구간 | 수십만 | ✅ **경로 엔진용** |

**결정**: **100m 격자 + 도로 엣지 병행**.
- 격자 → 모델 학습
- 도로 엣지 → 경로 엔진 inference 재사용

**시간 차원 추가**:

```
한 row = (격자 ID, 시간대 ID)
시간대 = {주간 06~18, 초저녁 18~21, 야간 21~01, 심야 01~06}
→ 60,000 격자 × 4 시간대 = 240,000 rows
```

### 4.5 샘플 수 요구량

| 모델 | 최소 샘플 | 권장 |
|---|---|---|
| Linear Regression | 수백 | ~1,000 |
| Random Forest | 수천 | ~10,000 |
| **LightGBM** | ~5,000 | **수만~수십만** |
| Deep Learning | 수십만 | 수백만 |

격자 기반 240,000 rows → LightGBM에 충분. **Feature의 질**이 양보다 중요.

### 4.6 Train/Val/Test Split — 공간 Split 필수

**함정**: Random split 쓰면 같은 동네 격자가 train과 val에 섞여 data leakage. 성능이 실제보다 과대평가됨.

**해결**: **자치구 기반 공간 split**.

```
Train: 20개 자치구 (관악, 서초, 강남, ...)
Val:   3개 자치구 (동작, 마포, 성동)
Test:  2개 자치구 (종로, 중구)
```

→ 모델이 "새로운 동네"에도 일반화되는지 정직하게 평가 가능.

## 5. 데이터 카탈로그 & 수집 전략

### 5.1 ⚠️ 검증 결과: 핵심 발견사항 (2026-04-10 실사)

공공데이터 포털 직접 검증 결과, **초기 가설에서 3가지 큰 변경**이 필요.

#### 🚨 변경 1: 안심이 CCTV 연계현황 사용 불가

- **원인**: OA-20923 "서울시 안심이 CCTV 연계현황"이 **2026-02-10자로 서비스 종료**됨 (국정원 지적, 안심주소 IP 노출 이슈). 자치구별 하위 데이터셋(OA-20924~20944)도 동일 리스크.
- **영향**: 좌표 기반 CCTV 개별 위치 데이터를 공개 소스에서 확보 불가. 원래 가장 중요한 feature로 기대했던 "반경 500m 내 CCTV 개수"의 계산 기반이 사라짐.
- **대응**:
  - 자치구 집계 CCTV(OA-2722, OA-2734, OA-21097)를 자치구 수준 feature로 사용
  - **OA-21695 "서울시 안심귀갓길 경로"** (362개, 좌표 있음)로 안전 인프라 feature 보강
  - 가로등·안심택배함·무인민원발급기·어린이보호구역 등으로 "감시 카테고리" 재구성
  - 빅데이터캠퍼스 방문 시 CCTV 좌표 데이터 확보 가능성 확인

#### 🎁 변경 2: 안심귀갓길 경로 데이터 발견 ★

- **데이터**: OA-21695 "서울시 안심귀갓길 경로" — **서울시가 이미 지정한 362개 안전 경로**, Shapefile + Excel, 공공누리 1유형, 활성
- **가치**:
  - 경로 추천 엔진에서 **baseline 경로**로 활용 가능
  - "서울시가 공식 지정한 안심 경로를 우선 사용하고, 사각지대는 모델이 보완"이라는 **강력한 발표 스토리**

## 9. 심사 대응 Q&A 준비

### Q1. "경로 탐색은 왜 ML이 아닌가요?"

> "경로 탐색의 본질적 어려움은 '엣지에 어떤 안전 점수를 매길 것인가'입니다. 이 부분은 LightGBM으로 학습했고, 그 결과물을 기반으로 Dijkstra가 수학적으로 최적 경로를 찾습니다. AI는 '판단'을 담당하고, 고전 알고리즘은 '탐색'을 담당하는 역할 분담 구조입니다."

### Q2. "딥러닝은 왜 안 쓰셨나요?"

> "Feature가 25개 수준의 tabular 데이터에서는 LightGBM 계열이 딥러닝보다 일관되게 우수합니다. 또한 LightGBM은 feature importance를 제공해서 '왜 이 점수가 나왔는지' 설명이 가능합니다. 안전 서비스는 설명 가능성이 핵심이라 이 선택이 더 적절합니다."

### Q3. "학습 데이터의 정답(y)은 어떻게 만들었나요?"

> "'진짜 안전 점수'라는 정답은 존재하지 않기 때문에, Hybrid 전략을 썼습니다. 학습에는 격자별 범죄 발생 건수를 proxy label로 사용하고, 검증은 팀원이 직접 500개 주소에 라벨링한 expert set으로 수행했습니다. 성능 metric은 모델 출력과 expert score의 Spearman 상관계수입니다."

### Q4. "Data leakage는 어떻게 막았나요?"

> "자치구 기반 공간 split을 적용했습니다. 같은 동네의 격자가 train과 validation에 섞이지 않도록, 자치구 단위로 20-3-2 분할했습니다. 이렇게 하면 모델이 처음 보는 동네에도 일반화되는지 정직하게 평가할 수 있습니다."
