## 1. 프로젝트 배경

학술 논문을 읽는 과정에서 발생하는 3가지 문제:

| 문제 | 설명 |
|------|------|
| **논문이 너무 많다** | 어디서부터 읽어야 할지 모른다 |
| **맥락 없이 읽는다** | 전문 용어, 배경 지식 부족으로 이해가 어렵다 |
| **읽은 게 쌓이지 않는다** | 논문 간 관계를 파악하기 어렵고, 읽은 내용이 체계적으로 정리되지 않는다 |

Deepen은 이 문제를 **챗봇 + 지식그래프 + 추천시스템**으로 해결한다.

---

## 2. 서비스 한 줄 정의

> 사용자가 논문을 탐색·분석하며 **자신만의 지식 그래프를 구축**하고, 추천을 통해 **그래프를 확장**해나가는 플랫폼

---

## 3. 핵심 사용자 플로우

```
[진입]  검색 / 추천 / 외부 링크로 논문 발견
  ↓
[확인]  논문 메타데이터 확인 + 외부 원문 링크로 이동
  ↓
[분석]  챗봇에 논문·질문·리소스 투입 → 구조화 데이터 생성
  ↓
[축적]  지식그래프에 노드·엣지 자동 추가 + 사용자가 직접 편집
  ↓
[확장]  추천시스템이 그래프 기반으로 다음 논문 제안
  ↓
  ↺    반복 — 그래프가 점점 성장
```

## 4. 3대 핵심 모듈

### 4-1. 챗봇 (분석 엔진)

논문을 이해하기 위한 **범용 분석 도구**.

| 항목 | 내용 |
|------|------|
| **인풋** | 자연어 질문, 논문 PDF, URL, 이미지, 기타 지식 리소스 — 형태 제한 없음 |
| **아웃풋** | 구조화된 분석 결과 (키워드, 개념, 방법론, 관계, 6-Layer 데이터) |
| **핵심 역할** | 비정형 입력 → 지식그래프에 넣을 수 있는 **정형 데이터**로 변환 |

**6-Layer 분석 프레임워크**: 논문을 6가지 관점에서 분해하는 Deepen의 고유 구조

| Layer | 이름 | 설명 |
|-------|------|------|
| 1 | Prior Work | 선행 연구 — 이 논문이 기반한 연구들 |
| 2 | Key Concepts | 핵심 개념 — 이해에 필요한 주요 용어·이론 |
| 3 | Pipeline | 방법론 — 논문의 실험·구현 과정 |
| 4 | Follow-ups | 후속 연구 — 이 논문 이후의 발전 방향 |
| 5 | Industry Use | 산업 적용 — 실제 활용 사례 |
| 6 | Open Questions | 미해결 문제 — 남은 과제와 연구 기회 |

### 4-2. 지식그래프 (Obsidian형 시각화)

읽은 논문과 개념이 자동으로 쌓이는 **개인 지식 지도**.

| 항목 | 내용 |
|------|------|
| **인풋** | 챗봇이 추출한 구조화 데이터 (자동) + 사용자 직접 편집 (수동) |
| **아웃풋** | 노드·엣지 그래프 뷰, 로드맵 뷰, 논문별 노트 |
| **핵심 역할** | 논문·개념·키워드 간 관계를 시각적으로 탐색하고 편집하는 인터페이스 |

주요 기능:
- **그래프 뷰**: Obsidian처럼 노드와 엣지로 지식 구조 시각화
- **로드맵 뷰**: 학습 경로를 순서대로 보여주는 뷰
- **노트**: 각 논문 노드에 연결할 수 있는 마크다운 노트
- **편집**: 노드 추가/삭제, 관계 수정, 그룹핑

### 4-3. 추천시스템

그래프의 빈 곳을 채울 논문을 제안하여 **그래프 성장을 유도**.

| 항목 | 내용 |
|------|------|
| **인풋** | 지식그래프 상태 (키워드, 논문 간 관계) |
| **아웃풋** | 추천 논문 리스트 (순위 + 추천 이유) |
| **핵심 역할** | 사용자의 현재 지식 맥락에 맞는 다음 논문 제안 |

알고리즘 접근:
- **룰베이스**: 인용 관계, 같은 저자, 같은 학회 등 명시적 관계 기반
- **키워드 기반 유사도**: 추출된 키워드 간 자연어 유사도 계산
- 두 방식을 조합하여 스코어링

## 5. 모듈 간 데이터 흐름

```
            ┌──────────────────────────────────┐
            │           사용자 입력              │
            │  (자연어, PDF, URL, 이미지 등)      │
            └───────────────┬──────────────────┘
                            ▼
                     ┌─────────────┐
                     │   챗봇       │
                     │ 비정형→정형   │
                     └──────┬──────┘
                            ▼
                  ExtractionResult (JSON)
                   ┌────────┴────────┐
                   ▼                 ▼
           ┌──────────────┐  ┌──────────────┐
           │  지식그래프    │◄►│  추천시스템    │
           │  시각화/편집   │  │  논문 제안    │
           └──────────────┘  └──────────────┘
```

| 연결 | 방향 | 전달 데이터 |
|------|------|-----------|
| 챗봇 → 그래프 | 단방향 | `ExtractionResult` — 추출된 노드·엣지가 그래프에 자동 추가 |
| 그래프 → 추천 | 단방향 | 현재 그래프 상태 (노드 목록, 키워드, 관계) → 추천 대상 결정 |
| 추천 → 그래프 | 단방향 | 사용자가 추천 논문을 채택하면 그래프에 새 노드 추가 |

## 6. 공통 데이터 스펙

3개 모듈이 공유하는 데이터 포맷. **챗봇이 생성하고, 그래프가 소비하고, 추천이 참조한다.**

```typescript
// ── 노드: 지식그래프의 구성 요소 ──
interface KnowledgeNode {
  id: string;
  type: 'paper' | 'concept' | 'keyword' | 'method';
  label: string;
  source: string;           // 출처 (논문 ID, 채팅 세션 ID 등)
  metadata?: {
    doi?: string;
    authors?: string[];
    year?: number;
    venue?: string;
    // type별 추가 메타데이터
  };
}

// ── 엣지: 노드 간 관계 ──
interface KnowledgeEdge {
  source: string;            // source node id
  target: string;            // target node id
  type: 'cites' | 'extends' | 'uses' | 'related' | 'prerequisite';
  weight: number;            // 관계 강도 0~1
}

// ── 챗봇 → 그래프로 전달되는 단위 ──
interface ExtractionResult {
  nodes: KnowledgeNode[];
  edges: KnowledgeEdge[];
  layers?: SixLayerData;     // 6-Layer 분석 (논문 분석 시)
}

// ── 6-Layer 분석 데이터 ──
interface SixLayerData {
  priorWork: LayerItem[];
  keyConcepts: LayerItem[];
  pipeline: LayerItem[];
  followUps: LayerItem[];
  industryUse: LayerItem[];
  openQuestions: LayerItem[];
}

interface LayerItem {
  title: string;
  content: string;
  relatedNodes?: string[];   // 관련 노드 ID 참조
}

// ── 사용자의 전체 지식그래프 ──
interface UserKnowledgeGraph {
  userId: string;
  nodes: KnowledgeNode[];
  edges: KnowledgeEdge[];
  roadmaps: Roadmap[];
  notes: Note[];
}
```

## 7. 지원 기능

핵심 모듈을 보조하는 기능들:

| 기능 | 설명 | 연관 모듈 |
|------|------|----------|
| **논문 검색** | OpenAlex API 기반 논문 메타데이터 검색 | 그래프, 추천 |
| **논문 상세 페이지** | 메타데이터 표시 + 원문 외부 링크 + 추천 논문 표시 | 그래프, 추천 |
| **노트** | 논문 노드에 연결 가능한 마크다운 노트 | 그래프 |
| **로드맵 뷰** | 그래프 내 학습 경로를 순서대로 시각화 | 그래프 |
| **사용자 계정** | 개인 그래프 저장·관리 | 전체 |

## 8. 저작권 대응 전략

논문 원문의 직접 가공·요약·표시는 저작권 위반 리스크가 있다.

| 하지 않는 것 | 하는 것 |
|-------------|---------|
| 사이트에서 논문 원문 표시 | 메타데이터 + 외부 원문 링크 제공 |
| 우리가 논문을 요약·가공 | 챗봇 **도구만 제공**, 논문 투입은 사용자 책임 |
| 논문 콘텐츠 DB 저장 | 사용자별 분석 결과만 저장 (개인 그래프) |

핵심 원칙: **"우리는 분석 도구를 제공할 뿐, 논문을 가공하지 않는다"**

## 10. 팀 R&R

| 담당 | 모듈 | 주요 과제 |
|------|------|----------|
| **건우** | 챗봇 (Agent) | 프롬프트 설계, AI 에이전트 구조, `ExtractionResult` 출력 파이프라인 |
| **원석** | 추출 엔진 (6-Layer) | 6-Layer 필드 스펙 확정, 키워드 정규화 규칙, 추출 일관성 확보 |
| **궁종** | 추천시스템 | 룰베이스 + 유사도 알고리즘 설계, 스코어링 기준, OpenAlex 활용 |
| **정현** | UI / 지식그래프 | 그래프 시각화, 노드·엣지 인터랙션, 로드맵 편집 UX |

각 담당자가 정의해야 할 것:

```
1. 인풋: 내 모듈이 받는 데이터의 형태와 출처
2. 아웃풋: 내 모듈이 내보내는 데이터의 형태와 소비처
3. 서비스 내 위치: 전체 흐름에서 내 모듈이 어디에 위치하는지
4. 필요한 것: 구현에 필요한 기술, 리소스, 다른 모듈과의 합의 사항
```
