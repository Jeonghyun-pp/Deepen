### 의논2: 키워드 추출 방식

> "키워드 추출은 논문 별로 그냥 미리 해놓는 거 아니야?"

새로운 관점: 대화 시점에 실시간으로 추출하는 게 아니라, **논문 추가 시점에 사전 인덱싱**.

제기된 반론:
- "그 많은 논문에 대해서 다 할수 있을지 모르겠어" — 논문 수 증가 시 스케일 우려
- "논문 이외의 지식베이스(concept, memo 등)는 어떻게 처리해야할지 더 애매하고"
- "애초에 논문을 추천하려면 키워드 추출이 되어 있어야 되는데?" — 추천 기능과 키워드 추출이 묶여있음

**상태: 미결** — 원석이 추출 방식 검토해오기로 함. 원석 의견 듣고 이어서 결정.

### 의논3: 1-hop 상한선 수치

## 2. 논문 추천 품질 기준 (신규)

논문 추천 시 사용자가 확인해야 할 항목들:

- [ ] 추천이 관련성 있게 되었나 (내가 관심있는 분야가 맞나)
- [ ] 추출된 키워드가 적절한가
- [ ] citation 수 노출 (인지도 파악)
- [ ] 학회/저널명 + accept 연도 노출 (venue + year)

→ 논문 카드 UI에 반영 필요.

## 3. 서비스 아키텍처 관련

### 좌측 논문 목록 — 확인 필요

> "좌측에 있는 논문들 뜨는거는 사용자가 업로드하는게 아니라 인터넷에서 긁어와서 논문 뜨게 할거죠?"

**결론 필요:** 현재 프로토타입은 사용자 업로드 기반. 인터넷 자동 크롤링/검색 결과 반영 여부는 별도 논의.

## 4. 제품 확장 아이디어 (브레인스토밍)

### Zotero 레퍼런스 모델

> "Zotero처럼 사용자가 논문을 넣어주면 자동으로 키워드 추출해서 관계 매핑하는 것도 아이디어"

- Zotero 인기 이유: 저장 → DB 별도 보관 → 모바일/멀티 디바이스 접근
- Deepen 적용 시: 논문 추가 → 자동 키워드 추출 → 그래프 관계 자동 매핑

### 외부 협업

> "학술도서관이나 데이터를 제공해주는 플랫폼에 연락해서 협업하는 개념"

- 프로토타입이 충분히 완성도 있으면 데이터 파트너십 가능성

### 논문 → 책/지식 전반으로 확장

> "논문에만 국한할 필요가 없다. 책도 추천 알고리즘이 리스트업 형식이라서, 저렇게 시각화해도 예쁘고 시장성도 좋을 것 같음"

- 핵심 서비스 정의: **추천 + 시각화 + 요약(챗봇)**
- 이 구조는 논문 외 책, 아티클, 강의 등 다른 지식 소스로 확장 가능
