### 1. Batch Approval — 확정

**문제**: §4.1의 `for` 루프 안에서 `await waitForApproval()`이 순차 블로킹. LLM이 한 턴에 approval-required tool call을 여러 개 내보내면 승인 카드가 하나씩 순차로 뜸.

| 항목 | 결정 |
|---|---|
| 묶는 범위 | 같은 턴의 모든 approval-required tool call을 한 카드로 (다른 종류도 묶음) |
| 부분 승인 | 개별 체크박스로 항목별 승인/거부 선택 가능 |
| 부분 거부 시 | 거부된 것만 빼고 나머지 실행, LLM에 "N번째 거부됨" 알림 |

**변경 영향**: §4.1 Agent Loop 의사코드 재설계, §5.4 승인 카드 UI에 batch 스펙 추가, §4.3 시나리오 B 수정

### 2. 그래프 컨텍스트 압축 — 확정

**문제**: §4.4의 `buildSystemPrompt`가 노드 수 기준(50/200)으로만 분기. 질문과의 관련성 무시.

| 항목 | 결정 |

| Phase 1 방식 | 키워드 매칭 + 1-hop 이웃 |
| 키워드 추출 | 토큰 부담이 없으면 LLM으로, 부담되면 규칙 기반 |
| 1-hop 상한선 | 설정함 (허브 노드의 이웃 폭발 방지) |
| 임베딩 기반 | Phase 3에서 도입 |

**구체적 변경**:
```typescript
// AS-IS

// TO-BE
function buildSystemPrompt(graphData: GraphData, userQuery: string): string
```

**변경 영향**: §4.4 System Prompt 재작성, 부록 Phase 배치 변경

## Part 2: 의논사항 — 확정

### 의논 1: 대화 히스토리 — 확정

#### 런타임 히스토리 관리

**방식 1(Sliding Window) + 방식 3(중요도 기반) 2단계 조합.**

매 턴 종료 시:

```
1단계 — 중요도 기반 (메시지 type으로 기계적 분류)

  | 메시지 유형              | 규칙               | 이유                              |
  |------------------------|--------------------|-----------------------------------|
  | 사용자 메시지             | 유지               | 대화 맥락의 핵심                     |
  | 에이전트 최종 답변 (answer)| 유지               | 사용자가 참고하고 있는 내용            |
  | tool_result             | 한 줄 요약으로 교체  | 토큰 가장 많이 차지. 실행 후 원본 불필요 |
  | tool_start              | 삭제               | "검색 중..." 상태 표시. 완료 후 무가치  |
  | approval_request + 응답  | 삭제               | tool_result에 이미 반영              |
  | ui_action               | 삭제               | 프론트 제어 명령. LLM 컨텍스트 불필요   |
  | error                   | 삭제               | 해결된 에러는 보관 불필요              |

2단계 — Sliding Window

  1단계 적용 후에도 총 토큰이 한계치 초과 시,
  오래된 턴부터 버림.
```

LLM 판별 불필요. `event.type` 기준 switch문으로 구현.

#### 대화 요약 노트화 (팀 피드백 — 한원석)

런타임 정리와 별개로, **대화 요약을 논문 옆에 노트로 영속 저장**하는 아이디어.

- 논문은 일회성이 아니라 사용자가 다시 보기 위해 정리하는 것
- 대화에서 나온 분석/정리 내용을 노트 노드로 그래프에 저장
- 다음 세션에서 시스템 프롬프트가 해당 논문의 노트를 참조 가능
- 런타임 정리(컨텍스트 초과 방지)와 조합 가능: 대화는 정리하되, 핵심 내용은 그래프에 남음

**Phase 1에 포함.** 대화 종료 시 에이전트가 핵심 내용을 메모 노드로 추가 제안 → 승인 카드 → 저장. 기존 `add_node`(메모) + `add_edge` tool 조합으로 구현.

참고: 컨텍스트 엔지니어링이라는 연구 분야가 있음. 추후 공부하면서 고도화 가능.

### 의논 2: 키워드 추출 방식 — 미결

추후 논의. 팀 내 의견:
- (기존) 사용자 질문에서 LLM으로 키워드 추출 → 토큰 절약 방향
- (한원석) 논문별로 키워드를 미리 추출해서 저장해두는 방식
- (박정현) 논문이 많아지면 전부 가능한지 불확실, 비논문 노드(메모/문서) 처리 방안 미정

---

### 의논 3: 1-hop 확장 상한선 — 미결

추후 논의. 팀 내 의견:
- (기존) 기존 `edge.weight` (0~1) 기준 top-K
- (한원석) 지금 정하기 이름. 유사도 검증 먼저 하고, 옵시디언을 레퍼런스로 연구한 뒤 재논의
- (박정현) 동의. 옵시디언 연구 후 결정
