# Deepen 강의안 마스터리 전략

> 작성일: 2026-04-13
> 범위: "강의안 완결 이해" 시나리오를 중심으로 한 제품 철학·원칙·경쟁 포지셔닝·전략 결정
> 상태: 전략 얼라인먼트 문서. 구현 계획은 별도 로드맵으로.

---

## 0. 문서의 목적

이 문서는 "빠른 성장 + 기업 협력" 논의에서 출발해 **Deepen의 결정적 시나리오(강의안 → 증명 가능한 이해)** 로 좁혀지는 과정에서 얻은 인사이트를 기록한다. 이후 구현 결정의 판단 기준으로 사용한다.

---

## 1. 결정적 시나리오

> **"수업 강의안이 너무 복잡하다 → Deepen에 업로드 → Deepen이 강의안 모든 내용을 노드로 분화하고, 이해 순서와 각각의 내용, 그리고 '여기까지만 공부하면 강의안 내용은 전부 아는 것'임을 확신하게 한다 → 학습"**

이 시나리오가 Deepen의 핵심 자산(그래프·로드맵·에이전트)을 단일 user journey에 정합적으로 사용한다. 경쟁 도구들이 약속하지 않는 **완결성**이 결정적 차별점.

### 1.1 시나리오가 강한 이유

- 3축(그래프 + 로드맵 + 에이전트 분해)이 하나의 흐름에 전부 동원됨 → 기능 정당화가 자연스러움
- **"여기까지 하면 끝"** 의 완결성 보장은 NotebookLM·ChatGPT가 정의상 못 주는 가치
- 타깃의 고통이 크고 지불 의사 명확 (시험 2주 전 대학원생·전문시험 응시자)
- C향 유입 훅으로도 강함 ("강의안 드롭 → 학습지도 완성" 데모 한 장)

### 1.2 시나리오가 성립하려면 필요한 3요소

시나리오의 약속("빠짐없이·순서대로·증명 가능하게")을 지키려면 다음 세 축을 **반드시** 구현해야 한다:

1. **완결성 증명 UX** — "다 커버했다"를 주장이 아닌 **사용자가 검증 가능한 상태**로
2. **Prerequisite DAG** — 저자가 정한 목차가 아닌 **개념 간 논리·교육적 의존성**의 지도
3. **이해 확인 루프** — "읽었다"가 아닌 **행동으로 증명된 이해**

이 셋은 선택이 아니다. 하나라도 빠지면 약속이 무너진다.

## 2. 3원칙 — 철학과 원칙

### 2.1 완결성 증명 UX

**철학**
> "믿어주세요"가 아니라 **"직접 확인하세요"**. Deepen은 신뢰를 주장하지 않고, 사용자가 신뢰를 **검증할 수 있는 도구**를 준다.

완결성은 LLM 품질 문제가 아니라 **추적 가능성(traceability) 문제**로 재정의한다.

**원칙**
1. **원문은 진실의 단일 소스다** — 노드가 원문보다 앞서지 않는다. 모든 노드는 원문 chunk로 역추적돼야 한다. 역추적 불가한 노드는 "Deepen이 추가한 해설"로 별도 마킹.
2. **침묵보다 미완을 드러낸다** — 87%일 때 "87%"라 솔직히 표시. 미매핑 블록을 숨기지 않는다.
3. **주장은 증거와 동시에 제시한다** — 완결성 배지 옆엔 항상 클릭 가능한 원문 증거.
4. **사용자의 반박 권한을 보장한다** — 잘못된 매핑을 사용자가 언제든 수정 가능, 시스템은 학습 신호로 받음.

**충돌 시 우선순위**
- UX 깔끔함 vs 추적 가능성 → **추적 가능성 우선**
- 속도 vs 완결성 → **완결성 우선**

### 2.2 Prerequisite DAG

**철학**
> 지식은 **순서가 있다**. 이 순서는 저자의 목차가 아니라 **개념 간 논리적 의존성**이다. Deepen은 목차를 복제하지 않고, 학습자 관점의 의존성 지도를 재구성한다.

**원칙**
1. **Prerequisite는 "언급"이 아니라 "이해 불가"의 관계다** — A를 이해하려면 B를 먼저 이해해야만 하는 경우에만 엣지.
2. **DAG은 불변식이다** — 사이클은 버그. 순환 정의는 하나를 axiom으로 끊는다.
3. **암묵적 전제를 명시화한다** — `assumedKnowledge`도 노드로 올린다.
4. **여러 학습 경로를 허용한다** — 같은 레이어의 노드는 순서 자유.
5. **학습자 경험이 우선한다** — 사용자가 엣지를 거부하면 그 판단을 존중.

**충돌 시 우선순위**
- 저자 순서 vs 학습 순서 → **학습 순서 우선** (저자 순서는 토글)
- 완전성 vs 정확성 → **정확성 우선** (애매한 엣지는 안 그림)

### 2.3 이해 확인 루프

**철학**
> "읽었다"는 이해의 증거가 아니다. 이해는 **행동으로만 증명**된다. Deepen은 학습자가 스스로에게 정직할 수 있는 거울을 제공한다.

**원칙**
1. **학습자가 스스로를 속이지 못하게 한다** — "이해했어요" 버튼만으론 부족. 능동적 출력 필요.
2. **실패는 처벌이 아니라 신호다** — 틀리면 prerequisite 역추적 경로 제안.
3. **확인은 노드 성격을 따른다** — 정의/정리/절차/개념/논증형별로 다른 check item.
4. **완주의 정의는 엄격하다** — "모든 노드 ≥1회 mastered" 이외는 완주 아님.
5. **장기 기억까지 책임지진 않는다** — Deepen의 약속은 **첫 이해의 완결**까지.

**충돌 시 우선순위**
- 사용자 편의 vs 이해 증명 → **이해 증명 우선** ("스킵" 없음, "나중에"는 허용)
- 자동화 vs 학습자 주도 → **학습자 주도 우선**

### 2.4 메타 원칙 — 세 축을 관통하는 하나

> **Deepen은 학습자를 대신해 공부해주지 않는다. 학습자가 "자기가 어디까지 이해했는지 정확히 아는 상태"를 만들어준다.**

- 완결성 증명 = 자료의 어디까지 커버했는지 **안다**
- Prerequisite DAG = 어디가 막혔을 때 어디로 가야 하는지 **안다**
- 이해 확인 루프 = 머릿속 어디까지 들어왔는지 **안다**

세 가지 모두 **"안다"를 만들어주는 도구**이지 **"해준다"**가 아니다. 이 경계가 NotebookLM·ChatGPT와의 결정적 차이.

### 2.5 세 원칙의 통합 불변식

> **"완주했다"** ≡ `coverage(원문→노드) == 100%` AND `∀ node: mastery == mastered` AND `DAG에 cycle 없음`

이 불변식을 DB 레벨에서 체크 가능하게 만들면, UI는 이 상태를 보여주기만 하면 된다. **약속의 무게가 코드로 강제됨**.

## 3. 엣지케이스 스트레스 테스트

각 원칙이 깨질 상황을 검토한 결과 — **핵심은 유지되지만 표현 공간이 넓어져야** 함을 확인.

### 3.1 완결성 원칙 확장

| 엣지케이스 | 해결 설계 |
|---|---|
| 원문 오탈자·모순 | `contradiction` 엣지 타입. 모순을 숨기지 않고 명시화. Deepen은 원문을 수정하지 않는다. |
| 수식·figure·표 비중 높음 | chunk 타입 확장: `text \| equation \| figure \| table`. Figure는 OCR+캡션+주변 텍스트의 복합 chunk. |
| 중복·반복 많음 | 커버리지는 추적성 지표. 정보량은 노드 수·DAG 복잡도로 별도 표시. |

**원칙 수정 사항**: chunk 모델을 **멀티모달**로 확장.

### 3.2 Prerequisite 원칙 확장

| 엣지케이스 | 해결 설계 |
|---|---|
| 저자가 직관→정의 순으로 설명 | 엣지 타입 이원화: `logical-prerequisite` + `pedagogical-order`. 사용자 토글. |
| 진짜 순환 정의(범주론 등) | 에이전트가 axiom 후보 제안 + 사용자 승인. `axiomatic_in_context` 플래그. |
| 학습자가 이미 일부 앎 | "이미 알아요" 버튼 → mastered 전이. 단 퀴즈 실패 시 정직하게 mastery 회수. |

**원칙 수정 사항**: prerequisite 엣지를 **단일 타입에서 이원 타입으로** 분화.

### 3.3 이해 확인 원칙 확장

| 엣지케이스 | 해결 설계 |
|---|---|
| 문과/인문학 "정답 없음" 개념 | `논증 재구성` check item 추가. 정답 판정이 아닌 "일관성·근거 인용" 평가. |
| 객관식 요행 | 복수 형식 필수 + 시간 간격 재검증. 한 번 통과로 mastered 아님. |
| 시험 범위 외 학습 제외 희망 | 로드맵 단계에서 학습 범위 선택 허용. 부분 완주 뱃지는 정직하게 "58/74" 표기. |

**원칙 수정 사항**: check item에 **논증형 추가**, mastery에 **재현성 축 추가**.

## 4. 경쟁 구도

### 4.1 1차 비교: 유니브 AI (`univai.co.kr`)

**프레이밍 교정**: 유니브 AI는 incumbent가 아닌 **동시대 얼리 플레이어**.
- 2024-06 설립, 팀 규모 최소, 시드 1억(프라이머) 1건, 앱스토어 평점 120건 수준
- "월 1.5억 매출"설은 2차 매체 주장
- Pro 9,900원/월 → **한국 학부 시장 WTP 참고점**

**시사점**: "정면 승부 회피" 논리가 아니라 **"독자 축 유지가 제품 건강에 낫다"**로 재구성. 같은 좁은 시장에서 경쟁자끼리 서로를 참조점 삼으면 고객이 아닌 경쟁자 기준으로 제품이 만들어진다.

**유니브 AI의 설계 한계** (Deepen 3원칙 기준)
- 파일 단위 사고(여러 자료 통합 없음)
- 원문 커버리지 지표 없음
- 퀴즈가 체크리스트(경로 피드백 없음)
- 개념 간 구조 없음
- "끝났다" 정의 없음
- 자동결제·환불 불만(znitas 리뷰) → 신뢰 기반 제품과 대비

### 4.2 2차 비교: NotebookLM — 진짜 경쟁 압력

**핵심 약점 8개와 Deepen의 지렛대**

| # | NotebookLM 약점 | 구조적 원인 | Deepen의 지렛대 |
|---|---|---|---|
| 1 | 학습자 상태 모델 부재 | **Assistant stateless** 패러다임. 학습자 지식 상태를 저장 안 함 | **처음부터 stateful 학습자 모델** — 가장 큰 구조적 해자 |
| 2 | 원문 커버리지 미증명 | Studio 산출물의 자유 생성과 추적성이 충돌 | Chunk→노드 명시 매핑. 유연성↔추적성 교환을 감수 |
| 3 | 소스 간 구조 얕음 | 노트북 = 컨테이너 모델, 소스는 리스트 | 그래프가 일급 객체 (연구 그래프 DNA) |
| 4 | 학습자 주도 경로 없음 | Assistant 패러다임(사용자 운전) | Tutor 패러다임(Roadmap 일급) |
| 5 | 소스 버전 관리 부재 | Static snapshot 설계 | Diff 기반 부분 재매핑 가능성 |
| 6 | API·상호운용 부재 | Google 생태계 lock-in 전략 | 열린 그래프, BYO key, 내보내기 |
| 7 | 프라이버시 인식 비용 | Google 서비스 전반의 신뢰 부채 | 로컬 우선 옵션, 자체 호스팅 |
| 8 | Audio Overview 얕음 | 엔터테인먼트 TTS와 정확성의 충돌 | "정확한 이해가 생명인 자료" 타깃 |

**따라올 수 있는 약점 vs 못 따라올 약점**
- **따라올 수 있음**: Audio 깊이, API, 프라이버시 인식 → 언제든 개선 가능. 공격해도 해자 안 됨
- **못 따라올 약점**: 학습자 상태·완결 증명·지식 그래프 → 제품 철학 전환 필요. Deepen의 해자 후보

### 4.3 진짜 경쟁 맵

유니브 AI 같은 국내 얼리 플레이어보다 다음 셋이 Deepen의 진짜 시험대:

1. **NotebookLM** — Google 무료·글로벌·리소스 압도
2. **ChatGPT Projects / Claude Projects** — 범용 LLM이 "충분히 좋음"에 도달하면 별도 도구 필요성 약화
3. **Anki / Obsidian** — 기존 습관의 전환 비용이 해자

## 5. 전략 결정

### 5.1 포지셔닝

| | 상대 |
|---|---|
| **유니브 AI**와의 대비 | "더 빨리" vs **"빠짐없이"** (속도 vs 완결) |
| **NotebookLM**과의 대비 | "자료를 더 잘 소비" vs **"자료를 완전히 이해했음을 증명"** (소비 vs 완주) |
| **ChatGPT 업로드**과의 대비 | "무엇이든 물어보는 창구" vs **"학습의 상태 있는 여정"** (stateless vs stateful) |

**상위 은유**
- NotebookLM = 오디오북·다이제스트 제조기 (정보 → 소비 형태)
- Deepen = 학습 커리큘럼 자동 생성기 + LMS (자료 → 완결 학습 과정)

### 5.2 핵심 약속

> **"여기까지 하면, 강의안은 정말 다 안 겁니다."**
> Deepen은 강의안을 쪼개 개념 그래프로 만들고, 이해의 순서를 찾아내고, 당신이 정말 이해했는지 마지막 한 노드까지 확인합니다.

### 5.3 타깃 재정렬

**피해야 할 시장**: 한국 학부 시험 벼락치기 (유니브 AI 포지션, 9,900원 가격대, 마진 박함)

**공략 시장 우선순위**:
1. 대학원 수업 (한 과목 누적 학습, 자료 복잡, 완결 중요)
2. 연구자 (논문 읽기, 연구 그래프 v2 원래 DNA와 일치)
3. 전문시험 (CFA·의사국시·변호사시험 등 "대충 요약 금지" 시장)
4. 사내 교육·컴플라이언스 (B2B, 지불 의사 높음)

**가격 권한**: NotebookLM 무료/$20 구간 아래로 내려갈 수 없음. **그 이상의 가치**로 올라가야 함.

### 5.4 B2B·협력 방향

- **대학 도서관·대학원 교육팀** — 연구방법론 수업 번들. "AI 에이전트 실습"은 비어 있는 채널
- **KISTI·연구재단·DBpia·RISS** — 공공 학술 데이터와 결합
- **제약/바이오 R&D** — 경쟁사 논문 모니터링 + 내부 지식 연결. Causaly가 잡은 시장이지만 가격 공백
- **출판사·학술 플랫폼** — "논문 페이지 → Deepen으로 열기" 버튼
- **학원·인강 업체** — 교재 → 학습지도 변환 B2B

단기는 **시드 그래프 템플릿 + 공개 공유**로 유기 성장 검증, 중기는 **한 개 분야(예: 바이오) 랩 파일럿**으로 레퍼런스 확보.

## 6. 아키텍처 원칙 — LLM과 코드의 책임 분리

### 6.1 근본 원칙

> **LLM은 제안(generate)하고, 코드는 검증(verify)한다.**
> 완결성·DAG 유효성·완주 판정 같은 "**약속**"은 결정론적 코드가 **최종 판정권**을 가져야 한다.

핵심 가치를 LLM에 맡기면 **확률적 증명**이 되고, "증명"이라는 단어 자체가 모순이 된다. 반대로 순수 rule-based는 자연어·멀티모달·도메인 다양성 앞에서 품질이 안 나온다. 따라서 이분법이 아니라 **책임 분리**로 간다.

> **LLM = Untrusted Worker. Code = Trusted Judge. DB = Invariant Guard.**

### 6.2 3원칙별 책임 분할

**완결성 증명**

| 역할 | 담당 | 내용 |
|---|---|---|
| 제안 | LLM | chunk → 노드 매핑 후보, 신규 노드 초안 |
| 검증 | 코드 | `unclaimed = allChunks - ∪(node.sourceChunks)` 집합 연산, 커버리지 % |
| 판정 | 코드 | "완결성 100%" 배지는 `unclaimed.isEmpty()` 일 때만 |

LLM이 "다 커버했어요"라 말해도 코드가 직접 집합 계산. LLM 주장은 무시.

**Prerequisite DAG**

| 역할 | 담당 | 내용 |
|---|---|---|
| 1차 필터 | 코드 | Syntactic pass(명시 참조), semantic pass(정의문 내 언급) — 파서·임베딩 유사도 |
| 제안 | LLM (judge pass) | "A가 B의 prerequisite인가?" 이진 판정 + 근거 |
| 검증 | 코드 | Cycle 감지(Tarjan SCC), topological sort, layer 계산 |
| 판정 | 코드 | 사이클 있으면 DAG 아님. LLM이 괜찮다 해도 거부 |

LLM의 역할은 엣지 후보의 **품질 향상**이지 그래프 **무결성 보장**이 아님.

**이해 확인 루프**

| 역할 | 담당 | 내용 |
|---|---|---|
| 제안 | LLM | check item 문항 생성, 서술형 채점 보조 |
| 검증 | 코드 | Cloze·순서배열·객관식은 정답 비교 |
| 상태 머신 | 코드 | `unseen → viewed → tested → mastered` 전이, 재검증 시점, 실패 시 prerequisite 역추적 |
| 불변식 | 코드 | `coverage 100% AND ∀ node mastered AND DAG acyclic` 평가, DB constraint로 강제 |
| 판정 | 코드 | "완주 뱃지"는 불변식 통과 시에만 |

LLM은 **논증형 채점** 같은 fuzzy 영역만. 이 경우에도 결과는 **사용자 재반박 가능 상태**로 남김.

### 6.3 아키텍처의 이점

- **감사 가능성** — 어떤 판정도 코드로 재현 가능
- **회귀 안전성** — LLM 모델·프롬프트 교체가 완주 판정을 흔들지 않음
- **테스트 가능성** — 핵심 로직을 LLM 없이 단위 테스트 가능
- **비용·속도** — 핵심 판정은 LLM 호출 없이 즉시
- **철학과 일치** — "검증할 수 있는 도구를 준다"가 아키텍처에 반영

### 6.4 설계 귀결

이 원칙을 지키려면 데이터 모델·파이프라인이 다음을 만족해야 한다:

1. **LLM 출력은 검증 가능한 스키마로** — 자유 서술 금지. `{chunkId, nodeId, confidence, justification}` 같은 구조. 스키마 위반 시 재시도.
2. **판정 함수는 pure function** — `isComplete(graph, chunks) → bool`, `isValidDAG(edges) → bool`, `isMastered(node, attempts) → bool`. 외부 의존 없음.
3. **LLM 결과는 "제안" 상태로 저장** — `proposed_edges`, `proposed_mappings` 테이블. 검증·승인 후에만 main 테이블로 이동.
4. **사용자 반박이 판정을 바꿀 수 있어야 함** — LLM 오판 시 사용자 수정 → 불변식 재계산 → 뱃지 변동.

### 6.5 타협 포인트

원문 chunk 자동 분할과 개념 추출은 LLM 없이 품질이 안 나온다. 이 단계에서 LLM이 **오염**시키면 뒤의 모든 검증도 의미 없다. 완화:

- **확정적 전처리 선행** — PDF 파서로 블록 단위(페이지·문단·수식) 분할. LLM 없이 확정적.
- **LLM은 의미적 그룹핑에만** — "이 세 문단이 한 개념" 같은 판단. 원자 chunk는 보존되고 그룹핑만 제안.
- **사용자가 최후 검수** — 커버리지 100%여도 "매핑 틀림" 반박 가능.

즉 **"LLM 배제"가 아니라 "LLM이 틀려도 불변식은 복구 가능"**하게 설계.

## 7. 구현 방향 — 3개 틈 모두 공략

**결정**: 학습자 상태 모델 + 완결 증명 + 지식 그래프 세 틈은 **분리 불가**. 하나라도 빠지면 약속이 무너지고, 셋이 맞물려야 불변식이 성립. 따라서 **3개 모두 MVP 범위**에 포함.

### 7.1 우선순위 (구현 의존성 기준)

의존 관계상 다음 순서로만 가능:

1. **SourceChunk 데이터 모델 + 추출 파이프라인 + 커버리지 배지**
   → 완결성의 기반. 없으면 나머지가 공허함
2. **Prerequisite edge type + syntactic pass + 레이어드 로드맵 뷰**
   → 학습 경로의 뼈대
3. **Check item 자동 생성 + mastery 상태 머신**
   → 이해 확인 루프의 최소 형태
4. **LLM judge pass + 실패 역추적 + 완주 뱃지**
   → 세 축의 통합 및 불변식 강제

### 7.2 데모 시나리오 (North Star)

대학원 1학기생, "확률적 프로그래밍" 강의안 87p:
- **0초**: PDF 드롭
- **90초**: 노드 54개 레이어드 DAG 생성. 커버리지 94%·미매핑 4 블록 표시
- **사용자 액션**: 미매핑 블록 클릭 → figure 2를 노드로 승인 → 커버리지 100%
- **5분**: Level 1 첫 노드 "확률 측도의 정의" 읽고 cloze 퀴즈 통과 → 노드 mastered
- **30분**: Level 3에서 "Bayesian updating" 퀴즈 실패 → "prerequisite 'conditional probability' 재검토" 역추적 제안
- **2일 후**: 마스터 52/54 → 완주 완료 배지: _"이 강의안(p.1~87, 54 개념)을 빠짐없이, 순서대로 이해했습니다."_

### 7.3 최소 검증 경로

MVP의 end-to-end 경로는 **"한 강의안 → 한 과목 완주"**. 기능을 넓히기보다 **하나의 완결된 루프**를 먼저 동작시킨다.

- 업로드 → 노드 생성 → 커버리지 표시 (완결성 최소)
- Level 1~2 DAG + 의존성 한 타입 (DAG 최소)
- Cloze 퀴즈 한 타입 + mastery 상태 (이해 확인 최소)
- 완주 불변식 체크 + 완주 뱃지

이 경로 하나가 돌면 나머지는 확장 작업.

## 8. 남은 결정 포인트

구현 시작 전 확정 필요:

1. **SourceChunk 데이터 모델 구체 스키마** — chunk ID 체계, 멀티모달 필드, 역참조 인덱스
2. **Prerequisite 엣지 추출 파이프라인** — syntactic / semantic / LLM judge의 3단 구체 설계
3. **Check item 자동 생성 규칙** — 노드 타입별 템플릿, 재생성·사용자 신고 루프
4. **완주 불변식의 DB 레벨 표현** — 상태 머신 구현 위치(DB trigger vs 애플리케이션)
5. **로컬 우선/클라우드 하이브리드 아키텍처** — 프라이버시 차별점 구현 여부·시점
