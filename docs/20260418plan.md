## 2. 설계 원칙 Q&A

### 2.1 노드 생성 기준

**정의**: 노드 = "마스터리 검증이 가능한 단위 개념". 전략 문서 §2.3 "이해 확인 루프"와 정합.

**노드로 만드는 것**
- **정의형**: "A is defined as...", "A란 ..." 같은 정의문이 명시된 용어
- **정리·결과형**: 명명된 정리·공리 (Bayes' theorem, Central Limit Theorem 등)
- **기법·절차형**: 명명된 알고리즘·방법 (EM algorithm, Backpropagation 등)
- **핵심 개념**: 섹션 내 3회 이상 반복되면서 설명 블록을 가진 용어

**노드로 안 만드는 것**
- 일회성 언급, 예시, 비유, 전환 문구
- 섹션 제목 자체 (제목은 섹션 구조로, 내용물이 노드)
- 자연어 부연설명만 있는 일반 명사

**Granularity 파라미터 (프롬프트에 고정)**
- 섹션당 최대 15 노드 — 폭발 방지
- 500~1000 단어당 1 노드 — 휴리스틱
- 모든 노드는 최소 1개 chunk에 매핑 필수 — 전략 문서 §2.1 추적성 원칙

**한계**: 이 기준은 LLM 하이퍼파라미터 성격이라 Week 2 첫 업로드 결과 보고 튜닝. 첫판 완벽 불가능.

### 2.2 PDF 처리 방식

LLM이 PDF를 직접 읽지 않는다. 2단계 분리. 전략 문서 §6 "LLM은 제안, 코드는 검증" 원칙 준수.

```
PDF 파일
   │
   ▼  [Stage 1: 결정적 코드, LLM 없음]
   pdfjs / unpdf로 텍스트 추출 → 문단 단위 chunk → DB 저장
   │
   ▼  [Stage 2: 휴리스틱, LLM 없음]
   페이지·제목 기반 섹션 묶음
   │
   ▼  [Stage 3: LLM]
   섹션 텍스트만 프롬프트로 전달 → 노드 후보 JSON 응답
```

**LLM이 보는 것**: PDF 파일이 아닌 추출된 텍스트 블록
**LLM이 못 보는 것**: PDF 레이아웃, figure 이미지, 렌더링된 수식

**수식·figure 처리**: 해당 chunk를 `equation_placeholder`로 마킹, LLM에 전달 skip. UI에서 "미지원" 명시. 멀티모달 확장은 North Star 논의 이후.

### 2.3 모델 선택 근거

| 태스크 | GPT-4o-mini 대비 오픈소스 품질 | 결정 |
|---|---|---|
| 노드 추출 | 드롭 5~15% | 베타 출시 후 Solar A/B 가능 |
| 에이전트 tool calling | 드롭 20~40% (multi-turn 불안정) | gpt-4o-mini 고정 |
| 한국어 전문용어 | Solar가 더 좋을 수도 | 추출 단계만 실험 |

베타 기간엔 모델 교체 없음. Provider 어댑터 도입은 프로덕트화 시점.

## 3. 주차별 실행 계획

각 주차 마지막에 **Go/No-Go gate**. 기준 미달이면 다음 주차 진입 지연.

### Week 1 — Foundation

**목표**: 로그인 → 빈 캔버스 → 수동 노드 1개 추가 → 새로고침 후 유지

**Tasks**
1. Supabase 프로젝트 연결, 로컬 env 셋업
2. Drizzle 스키마 작성
   - `users` (auth.users 확장), `sessions`, `documents`, `chunks`, `nodes`, `edges`, `chunk_node_mappings`
3. Row-Level Security 정책 (`user_id = auth.uid()`)
4. `/login` magic link 페이지 + 미들웨어로 `/graph` 보호
5. `GET /api/graph/current` — 현재 사용자 그래프 조회
6. `POST /api/nodes`, `POST /api/edges` — mutation 엔드포인트 (세션 검증)
7. `app/graph/GraphShell.tsx:31` `sampleGraphData` import 제거 → API fetch
8. `useGraphStore.initData()` 리팩터, 낙관적 업데이트 정책 결정
9. `EmptyGraphState` 컴포넌트 + 임시 "수동 노드 추가" 버튼 (Week 2 전 검증용)

**Gate**
- [ ] magic link 로그인 동작
- [ ] 빈 그래프 렌더링
- [ ] 수동 노드 1개 추가 → 새로고침 후 유지
- [ ] 다른 계정으로 로그인 시 내 노드 안 보임 (RLS 확인)

**리스크**: RLS 정책 실수로 교차 데이터 노출 → gate에서 2계정 교차 테스트 필수

---

### Week 2 — Document Pipeline

**목표**: PDF 드롭 → 90초 내 그래프에 노드 10+ 생성, 클릭하면 출처 chunk 보임

**Tasks**
1. Dropzone UI (캔버스 내 + `/upload` 경로 둘 다)
2. `POST /api/documents/upload` — Supabase Storage에 PDF 저장
3. PDF 파싱 (`unpdf` 또는 `pdfjs-dist`) — 텍스트 + 페이지 경계 보존
4. **Stage 1 (결정적)**: 문단 단위 chunk 분할, 페이지 좌표 기록 → `chunks` 테이블. LLM 없음
5. **Stage 2 (휴리스틱)**: 페이지·제목 기반 섹션 묶음
6. **Stage 3 (LLM)**: 섹션별 병렬 MAP 호출 (최대 5 동시, gpt-4o-mini, 구조화 JSON)
   - 기존 `lib/agent/tools/extract_concepts.ts` 재활용
   - 출력: `{nodes: [{label, type, tldr, sourceChunks}], edges: [...]}`
7. `chunk_node_mappings` insert — 추적성의 물리적 구현
8. Supabase Realtime으로 진행률 UI 실시간 업데이트
9. 수식·figure chunk는 `type: equation_placeholder`로 마킹만

**Gate**
- [ ] 본인 강의안 PDF 업로드 → 90초 내 노드 10+ 생성
- [ ] 업로드 진행률 표시
- [ ] 노드 클릭 시 RightPanel에 출처 chunk 표시
- [ ] 수식·figure 섹션이 "미지원" 뱃지로 솔직히 표시

**리스크**
- 수식 많은 PDF — Week 2 Day 1에 발견되면 최대 3일 지연. 대응: 첫 업로드는 수식 적은 것부터
- LLM 레이턴시 — 병렬 `Promise.all(concurrent=5)` 필수

---

### Week 3 — Agent Server Integration + UX Polish

**목표**: 에이전트 챗 → 서버 mutation → DB → 실시간 UI. 출처 추적 UX 완성.

**Tasks**
1. `lib/agent/runner.ts` — `add_node`, `add_edge` 툴을 서버에서 DB 쓰기로 변경. approval flow 유지
2. `lib/agent/prompt.ts` — 하드코딩 카운트 제거, DB query로 교체. 키워드 기반 context 주입 유지 (Phase 2는 나중)
3. 노드 중복 방지 dedup (label 정규화 + 간단 편집거리 매칭)
4. RightPanel → Graph 탭에 "출처" 섹션 (chunk 원문 스니펫 + 페이지 번호)
5. `/documents` — 업로드 이력, 삭제 가능 (cascade delete로 관련 노드·엣지·chunk 제거)
6. Empty/error/loading state 전반 정리
7. Onboarding 툴팁 — 첫 로그인 시 3단계
8. 사용자별 토큰 소비 로깅
9. Sentry free tier 연결

**Gate**
- [ ] 챗으로 "X를 Y의 prerequisite로 추가" → 승인 → DB 반영 → UI 즉시 업데이트
- [ ] 노드 클릭 시 출처 chunk 뜸
- [ ] 새로고침해도 챗 결과 유지
- [ ] 업로드 히스토리에서 문서 삭제 시 관련 노드 cascade 삭제

**리스크**
- Approval SSE 흐름과 서버 mutation의 race condition — 기존 테스트 시나리오 5개 smoke 필수
- Whiteboard localStorage ↔ DB nodes 동기화 엣지케이스 주의

### Week 4 — Deploy + Dogfood + Beta Invite

**목표**: 10명 초대 발송, 48시간 내 3명 이상 로그인

**Tasks**
1. Vercel 프로젝트 연결 + 환경변수 (Supabase URL·anon key·service key, OpenAI 키)
2. Supabase 프로덕션 프로젝트 분리 (dev DB와 격리)
3. 초대 토큰 시스템 — `/invite/:token` 경로, 화이트리스트 이메일만 가입
4. Usage cap 안전장치 — 월 토큰 한도 1인당 $10 상당 방어벽
5. Landing page 수정 — 클로즈드 베타 안내, 초대 없으면 wait list
6. 프라이버시 문구 한 단락 (PDF 저장·LLM 전달 명시)
7. 본인 dogfooding 3~4일 — 강의안 3개로 실제 학습 시도
8. 발견 버그 치명도 분류, Blocker만 수정
9. 10명 초대 발송 + 피드백 폼 (Tally 무료)
10. 첫 72시간 모니터링 — 에러·업로드 수·챗 턴 수

**Gate**
- [ ] 외부 네트워크에서 프로덕션 URL 접근
- [ ] dogfooding 중 치명 에러 0
- [ ] 10명 발송 완료
- [ ] 48시간 내 3명 로그인

**리스크**
- Supabase 무료 storage 1GB — 3~4개 강의안 × 10명이면 근접. Pro tier($25) 필요 여부 Week 3 말에 결정
- 10명 중 실사용 0명 가능성 — 초대 문구·onboarding 부족일 수 있음. Week 5 연장 옵션 확보

## 6. B 스코프에서 의도적으로 제외한 것

North Star 논의 이후로 연기. 현재 단계에선 구현하지 않음.

- **완결성 배지**: `unclaimed = allChunks - ∪(node.sourceChunks)` 계산 UI
- **Mastery 상태 머신**: `unseen → viewed → tested → mastered` 전이
- **Check item 자동 생성**: cloze·순서배열·MCQ
- **Prerequisite 엣지 자동 추출**: syntactic + semantic + LLM judge 3-pass. 현재는 `relatedTo`만
- **DAG 불변식 체크**: cycle 감지·topological sort
- **멀티모달 PDF 이해**: 수식·figure 이미지를 LLM에 직접 전달
- **임베딩 기반 중복 제거** (Stage 4)
- **BYO key UI**
- **Whiteboard localStorage → DB 마이그레이션** (현재 분리된 상태 유지)
