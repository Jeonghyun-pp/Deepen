# 고민거리 — Agent Evolution Plan의 3대 쟁점

> 작성일: 2026-04-13
> 출처: `agent-evolution-plan.md` / `agent-evolution-roadmap.md` 에 대한 `agent-evolution-plan-feedback.md`(건우) 지적을 구조화
> 상태: Phase 0~1 착수 전 확정 필요한 설계 고민 3건

## 쟁점 1 — DB 스키마

### 다루는 쟁점

표면: "Postgres냐 SQLite냐, RDB냐 그래프DB냐"
**실제 쟁점**: *Phase 0에서 내리는 스키마 결정이 Phase 2~8까지 전부 제약한다. 그 결정을 얼마나 신중하게 할 것인가.*

DB 선택은 드라이버 교체로 되돌릴 수 있지만, 스키마 모양은 한 번 데이터가 쌓이면 마이그레이션이 고통. 피드백이 찌른 지점은 여기.

### 우려되는 점

1. **노드/엣지 버전 관리 부재로 Phase 6 블로킹** — 단순히 `nodes(id, label, ...)`로 시작하면 체크포인트·undo 구현 시 전 테이블을 `nodes_history`로 쪼개야 함. `created_at`, `deleted_at`, `version`, `tx_id` 처음부터 고민 필요.
2. **`proposed` vs `confirmed` 경계 누락** — `strategy-lecture-mastery.md` §6.4가 요구하는 "LLM 제안은 별도 상태로 저장"이 스키마로 강제되지 않으면 승인 UX(Phase 4)와 완결성 불변식이 뒤에서 무너짐.
3. **chunk↔node 역매핑 인덱스 누락** — 커버리지 배지의 근간. 나중에 추가하면 백필 필요, 고아 chunk 정제가 큰 일.
4. **엣지 타입 확장성** — 현재 similarity 한 종. 마스터리 전략은 `logical-prerequisite` + `pedagogical-order` + `contradiction` + `assumed-knowledge` 등 최소 4~5종 요구. enum으로 박으면 마이그레이션, JSON meta에 숨기면 인덱스 못 붙임.
5. **SQLite 특화 기능 의존** — `json_extract`, `INTEGER` 시간 타입 등이 Postgres 전환(Phase 8) 시 발목.
6. **벡터 저장 위치 결정 지연** — Phase 2 임베딩 시 sqlite-vec로 같은 DB? 별도 서비스? 나중에 결정하면 하이브리드 검색 재설계.
7. **멀티 테넌트 경계** — `user_id` 컬럼 생략하면 나중에 전 테이블 백필. 지금 넣는 비용은 0.

### 추천 방법

**① 스키마 리뷰를 Phase 0 착수의 게이트로 (하루~이틀)**
- 엔티티 목록 고정: `nodes, edges, chunks, chunk_node_map, proposed_*, sessions, messages, tool_calls, snapshots, users`
- 각 테이블 기본 컬럼: `id, created_at, updated_at, deleted_at, user_id`
- 변경 이력 필요 테이블은 `version + tx_id` 결정

**② "Write-ahead" — 미래 Phase 산출물을 스키마에 미리 비워둠**

| Phase | 스키마에 미리 반영 |
|---|---|
| 2 임베딩 | `nodes.embedding BLOB NULL` |
| 4 승인 | `proposed_edges`, `proposed_mappings` |
| 6 체크포인트 | `tx_id` FK, `snapshots(tx_id, created_at)` |
| 마스터리 3원칙 | `chunks`, `chunk_node_map`, `node_mastery_state` |

**③ 첫 주에 이중 DB 마이그레이션 테스트** — Drizzle schema.ts → SQLite 적용 → 같은 파일로 Postgres 적용. 안 되면 이미 SQLite 특화 문법 의존 중.

**④ 불변식을 DB 제약으로** — FK·UNIQUE·NOT NULL 꼼꼼히. 완주 불변식은 CHECK·트리거로 일부 강제 가능.

**⑤ ADR 1장 남기기** — `docs/adr/0001-schema-v1.md`에 선택·대안·전환 트리거 기록.

### 한 줄 요약
스키마는 Phase 0의 작업이 아니라 **Phase 0~8 전체의 계약**. 착수 전 게이트 하루 + 미래 산출물 컬럼 수준 선제 선언 + 첫 주 이중 DB 마이그레이션 테스트, 세트로.

## 쟁점 2 — 히스토리 압축

### 다루는 쟁점

표면: "30턴 루프 비용이 감당 가능한가, $2/세션 상한이 맞나"
**실제 쟁점**: *루프 길이가 아니라 **입력 토큰의 누적 곡선**이 비용을 결정한다. 압축 로직의 품질이 이 곡선의 기울기를 정한다.*

가드레일은 사후 차단기. 차단되는 순간 UX는 이미 망가짐. 진짜 설계 대상은 **차단이 걸리지 않도록 입력을 평탄하게 유지하는 압축 전략**.

### 우려되는 점

1. **O(n²) 토큰 폭발** — 압축 없으면 누적 입력 토큰이 턴 수의 제곱. 10턴 $5, 20턴 $20. 30턴 도달 전 상한 강제 종료 → Phase 1 목적 무력화.
2. **도구 결과가 진짜 폭탄** — `search_papers_openalex` 같은 도구가 한 번에 수 KB JSON. 30턴 중 20턴이 도구 호출이면 **대화 이력보다 도구 결과가 10배 큼**. "메시지 단위" 압축만 보면 놓침.
3. **요약 품질 저하 → 루프 품질 저하** — 모델이 "이 노드는 이미 거부됨" 같은 세부 결정 잊음. 같은 제안 반복.
4. **조용한 비용 누수** — 압축이 "작동하는 것처럼 보이지만" 중복·불완전 절단 시 토큰은 조용히 늘고 가드레일 전까진 티 안 남. **관측 가능성 없으면 원인 추적 불가**.
5. **세션 경계 넘는 히스토리** — 대화→그래프 영속화(0410 의논1 확정)로 과거 세션 요약이 현재 세션 입력에 합류. 세션 내/간 경계 둘 다 설계 필요.
6. **캐시 무효화** — 프롬프트 캐시는 prefix 완전 동일이 전제. 매 턴 요약을 새로 쓰면 hit 0%.
7. **멀티 프로바이더/모델 교체** — Phase 7(Opus+Haiku)에서 압축 포맷이 한 모델 특화면 교체 비용 큼.

### 추천 방법

**① 압축을 "레이어 3종"으로 설계**

| 레이어 | 대상 | 방식 |
|---|---|---|
| A. 도구 결과 요약 | tool message | 결정적 구조 추출(JSON → 핵심 필드). LLM 요약 **아님** |
| B. 오래된 턴 요약 | assistant/user 페어 | n턴 초과 시 LLM 요약 치환, 원본은 DB |
| C. 상태 스냅샷 | 누적 결정·거부 | 매 턴 갱신 "작업 상태 블록" |

A가 효과 가장 크고 가장 쉬움. A부터.

**② 관측 가능성을 압축과 동시 출시** — 매 턴 입력 토큰 / 누적 비용 / 원본 대비 압축률 기록. 압축 버그는 조용히 온다.

**③ 회귀 테스트** — 30턴 고정 시나리오 → 총 입력 토큰 임계값 이하 CI 검증. "같은 도구 결과 2번 주입" 극단 케이스 스냅샷.

**④ 도구 응답 스키마에 "LLM용 요약" 필드 강제** — `{ full_result, summary_for_llm }` 분리. 런타임 기본값 summary만 히스토리 주입, full은 DB.

**⑤ 캐시 친화적 prefix** — 시스템 프롬프트 + 불변 배경 → 앞에. 요약·상태 스냅샷 → 뒤에. 압축 갱신해도 prefix 캐시 유지.

**⑥ 에이전트가 압축 자각** — `[이전 12턴 요약]` 명시. 세부 필요 시 `read_history(turn_range)` lazy load. Auto-context(Phase 2)와 맞물림.

**⑦ 비용 상한은 2단 경보** — 70% UI 경고 + 100% 강제 종료. 단일 하드 컷은 UX 거침.

### 한 줄 요약
루프 길이가 아니라 **입력 누적 곡선**이 비용을 정한다. 압축 3레이어 + 관측/CI 동시 출시 + 캐시 친화적 prefix 세트를 Phase 1 착수 전 확정.

## 쟁점 3 — 시나리오 현실성

### 다루는 쟁점

표면: "'50개 추가 조사'는 사용자가 실제로 하는 말이 아니다"
**실제 쟁점**: *예시 시나리오의 형태가 에이전트 설계의 모양을 정한다. 숫자 기반으로 설계하면 '얼마나'는 잘 풀지만 '언제 끝'은 못 푸는 에이전트가 나온다.*

시나리오는 홍보 문구가 아니라 **종료 조건·도구 선택·프롬프트 지침의 모양을 거꾸로 강제**. 3개가 전부 숫자형이면 세 축이 모두 숫자 의존으로 굳어짐.

### 우려되는 점

1. **"충분함 판정"이 내재되지 않음** — "50개 추가"는 카운터 하나로 종료. "attention 쪽 보강"은 **상태 → 빈약함 측정 → 충분함 판정** 3단 추론 필요. 전자로만 설계하면 후자가 영영 못 돌아감.
2. **`find_gaps`가 장식용** — 시나리오가 숫자형이면 호출될 이유 없음. 구현돼도 죽은 코드.
3. **실사용 시 "무한 추가"** — "보강해줘"에 종료 조건 없으면 가드레일에 걸려 멈춤. 사용자 눈엔 "막힌 응답". 신뢰 훼손.
4. **Stateful 차별점과 단절** — `strategy-lecture-mastery.md`의 stateful 해자는 "그래프 상태 읽고 부족한 곳 찾기". 숫자형 시나리오는 이걸 안 씀. 전략 문서와 **어긋남**.
5. **벤치마크는 여전히 필요** — "보강해줘"는 성공 판정 주관적, CI에 못 넣음. 숫자형은 테스트용으로 가치 있음. 피드백이 이걸 놓침.
6. **사용자 발화 데이터 부재** — 근거가 건우 직관 1개. 실제 수집된 발화 없으면 새 시나리오도 추측.
7. **UI 유도 가능성** — UI에 "N개 조사" 입력이 있으면 행동이 쏠림. 시나리오 교정만으론 부족.

### 추천 방법

**① 시나리오를 "사용자형 vs 벤치마크형" 2-트랙 분리**

| 트랙 | 목적 | 예시 | 성공 판정 |
|---|---|---|---|
| 사용자형 | 실제 UX 설계 기준 | "attention 쪽 빈약한데 보강해줘" | 수동 평가 |
| 벤치마크형 | 회귀 테스트·CI | "NLP 논문 20개 추가" | 결정론적 |

**② 사용자형 시나리오를 상태+명령 쌍으로**
```
상태: attention 노드 3개, RLHF 노드 0개
발화: "attention 쪽 빈약한데 보강해줘"
기대 궤적: find_gaps → search_papers → (제안 N개) → 사용자 승인
종료 조건: find_gaps "갭 없음" 반환 OR 사용자 승인 임계 도달
```
종료 조건·도구 순서·프롬프트 지침이 자동 도출.

**③ "충분함 판정"을 일급 시스템으로**
- **측정**: `find_gaps(area)` — 커버리지·밀도·외부 기준 대비 부족분 수치화
- **임계**: system prompt에 "임계 이하 반환 시 `done`" 룰 명시
- **브레이크**: 3턴 연속 갭 수치가 줄지 않으면 강제 종료 + 수렴 실패 신호 주입

**④ 사용자 발화 데이터 최소 1회 수집** — 팀 3~5명 × 5개 발화 = 30개 샘플. 추측 대신 근거 1g.

**⑤ `find_gaps` 품질을 시나리오와 함께 검증** — 사용자형 시나리오 각각에 대해 ground truth 수동 작성. 도구 품질 수치 추적.

**⑥ UI에 상태 기반 프리셋** — placeholder를 "빈약한 영역 보강 / 연결 정리 / 최신 흐름 추가"로. 사용자 발화 유도.

**⑦ 에이전트가 종료 조건 선언 강제** — 첫 턴에 "작업 이해 + 종료 조건"을 명시적으로 출력. 매 턴 self-check. "무한 추가" 방어선.

### 한 줄 요약
시나리오 형태가 종료 조건·도구·프롬프트를 거꾸로 강제. 2-트랙 분리 + 충분함 판정 시스템 승격(`find_gaps` + 임계 + 수렴 실패 브레이크) + 실제 발화 표본 수집, Phase 1 착수 전 고정.

## 통합 — 3개 쟁점이 공유하는 패턴

세 쟁점 모두 같은 구조: **플랜은 구조를 설계했고, 피드백은 그 안의 한 줄이 진짜 병목임을 짚는다.**

| Phase | 플랜의 초점 | 실제 병목 |
|---|---|---|
| 0 | "DB 쓰자" | **스키마 설계** |
| 1 | "30턴으로 늘리자" | **히스토리 압축** |
| 시나리오 | "50개 조사" | **충분함 판정** |

Phase 착수 전 이 세 병목을 **설계 수준에서 먼저 못 박아야** 뒤의 모든 Phase가 흔들리지 않음.
