### Q. 그러면 phase 0에서 rdb를 쓰는건 맞는거 같아?

맞아. 근데 "왜 맞는가"를 조금 더 정확하게 짚어볼게.

---

## 핵심 질문: Deepen이 저장하는 데이터가 뭔가?

세 종류가 있어.

**① 관계형 데이터** — 대화 히스토리, 승인 상태, 도구 호출 이력, 버전 이력
이건 명백히 RDB야. 행/열 구조가 자연스럽고, "이 세션의 승인 대기 항목들" 같은 쿼리를 SQL로 바로 쓸 수 있어.

**② 그래프 데이터** — 노드, 엣지
여기서 "그래프 DB 써야 하지 않나?" 라는 질문이 나오는데, 판단 기준은 **어떤 쿼리를 얼마나 자주 쓰느냐**야.

그래프 DB(Neo4j 등)가 압도적으로 유리한 건 **수십 홉 순회**가 실시간으로 필요할 때야. 예를 들어 "이 논문에서 출발해서 6단계 이내로 연결된 모든 저자를 찾아라" 같은 거.

Deepen의 실제 쿼리를 보면:
- 이 노드의 1-hop 이웃 가져와 → SQL `JOIN` 한 번
- 두 노드 최단 경로 → Phase 1~2 수준 그래프에선 재귀 SQL로 충분

연구 그래프 특성상 깊은 홉 순회가 핵심 기능이 될 가능성이 낮아. 논문 간 관계는 보통 1~2홉으로 의미 있는 결론이 나오거든.

**③ 벡터 데이터** — 임베딩 (Phase 2)
이건 sqlite-vec 확장으로 SQLite 안에서 처리해. 별도 벡터 DB를 쓰는 게 성능은 더 좋지만, 지금 규모(수천~수만 노드)에서는 오버킬이고 스택이 복잡해져.

---

## 결론

지금 Deepen 규모와 쿼리 패턴에서는 **SQLite + Drizzle이 맞는 선택**이야.

단, 한 가지 전제가 있어. "나중에 Postgres로 갈아탈 수 있도록 Drizzle로 추상화해둔다"는 문서의 전략이 실제로 지켜져야 해. 스키마를 SQLite 특화 기능에 의존하게 짜버리면 나중에 마이그레이션이 고통스러워져. 그게 Phase 0의 진짜 리스크야 — DB 선택이 아니라 **스키마 설계**.

### Q. 지금 왜 postgresql을 안 쓰고 sqlite를 쓰지? postgresql이 낫지 않나?

좋은 질문인데, "낫다"의 기준이 뭐냐에 따라 달라져.

---

## PostgreSQL이 SQLite보다 나은 것들

- 동시 접속 처리 (멀티 유저)
- 운영 환경 안정성
- pgvector (Phase 2 임베딩을 더 잘 처리)
- 대용량 데이터

이건 맞아. 장기적으로 Deepen이 서비스가 되면 PostgreSQL이 맞는 선택이야.

---

## 근데 지금 단계에서 SQLite를 쓰는 이유

한 줄로 말하면 **"지금 Deepen은 혼자 쓰는 로컬 도구"**야.

PostgreSQL을 쓰려면 서버를 띄워야 해. 로컬에서 개발할 때 Docker로 PostgreSQL 컨테이너 올리고, 환경변수 설정하고, 연결 관리하고 — 이 오버헤드가 지금 단계에서는 순수 낭비야. SQLite는 파일 하나야. 설치도 없고, 서버도 없어.

Drizzle ORM을 쓰는 이유가 여기 있어. 스키마를 Drizzle로 짜두면 나중에 PostgreSQL로 전환할 때 드라이버만 바꾸면 돼. SQLite 특화 문법에 의존하지 않는 한.

---

## 그래서 판단 기준은

| 상황 | 선택 |
|---|---|
| 지금 — 혼자 개발, 로컬 실행 | SQLite |
| Phase 6 완료 후 — 다른 사람도 쓴다 | PostgreSQL 전환 |
| Phase 8 — 백그라운드 워커 병렬 실행 | PostgreSQL 필수 |

SQLite의 치명적 약점이 **파일 동시 쓰기**인데, 혼자 쓰는 동안은 이게 문제가 안 돼. 백그라운드 에이전트 여러 개가 동시에 DB에 쓰기 시작하는 Phase 8부터는 SQLite로 못 버텨.

지금 SQLite를 쓰는 건 "PostgreSQL보다 못한 선택"이 아니라 **"지금 필요한 것만 쓰는 선택"**이야.
