## phase 0 - rdb가 맞는지? 
그래프 데이터를 rdb에 저장하는거면 그래프db가 더 낫지 않은지 검토하면 좋을듯. 문서에서 적었듯 db 스키마를 처음 잘못 설계하면 나중까지 계속 고생하니 많은 고민과 심혈을 기울여 설계 잘해야함
빠른 시작보다 스키마 설계에 시간을 더 써야 하는 구간이야.

## phase 1 - 비용이 감당 가능한가?
계산해보면, gpt-4o 기준으로 30턴 루프에서 턴마다 입력 4k 토큰 + 출력 500 토큰이면 한 세션에 약 $0.5~1.0야. 문서가 기본 상한으로 걸어둔 $2/세션은 꽤 넉넉한 편이고, 이걸 초과하면 가드레일이 루프를 강제 종료해.
진짜 비용 리스크가 오는 건 두 가지 경우야. 하나는 히스토리 압축 로직이 망가졌을 때 — 매 턴마다 전체 대화 이력을 다시 집어넣으면 턴이 쌓일수록 입력 토큰이 폭발함. 둘은 Phase 8(background agent) — 수백 턴이 백그라운드에서 돌기 시작하면 세션당 비용이 다른 차원이 돼.
즉 Phase 1 자체보다 히스토리 압축(Phase 1 작업 6번)을 제대로 구현하는 게 비용 관리의 핵심이야. 압축이 잘 되면 30턴도 $1 안에 들어오고, 압축이 망가지면 10턴에 $5 넘어갈 수 있어.

## 시나리오 현실성 문제
Q. 실제로 논문 조사하면서 저렇게 질문하나? "Transformer 이후 중요한 NLP 논문 10개 추가 조사해서 내 그래프에 붙여줘" 가 더 현실적이지 않으려나?

건우 지적이 맞아. "50개 추가 조사해줘"는 실제 사용자가 하는 말이 아니야. 연구할 때 논문 개수를 미리 정하고 탐색하는 사람은 거의 없거든.
현실적인 사용 시나리오를 다시 써보면 이런 느낌이야.
* "Transformer 이후 중요한 NLP 흐름 좀 채워줘" — 개수가 아니라 커버리지로 지시
* "내 그래프에서 attention 쪽이 너무 빈약한데 보강해줘" — 현재 그래프 상태 기반
* "RLHF 관련 논문들 연결고리 정리해줘" — 기존 노드들 사이의 관계 정제
이게 중요한 이유는 에이전트 루프 설계에 영향을 미쳐서야. "50개"라는 숫자가 명확하면 루프 종료 조건이 단순해(50개 추가하면 done 호출). 근데 "중요한 흐름 채워줘"라고 하면 에이전트가 스스로 "이 정도면 충분하다"를 판단해야 해 — 이건 훨씬 어려운 문제야. done을 언제 호출할지에 대한 system prompt 지침이 더 정교해져야 하고, find_gaps 같은 도구가 실질적으로 더 중요해져.
문서의 시나리오 3개가 구체적인 숫자 중심으로 쓰인 게 오히려 구현 편의를 위한 단순화일 가능성이 높아. 실제 사용자 행동 기반으로 시나리오를 다시 정의하는 게 맞아.
